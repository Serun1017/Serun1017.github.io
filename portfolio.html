1. ACV: Azure 기반 허위정보 실시간 차단 및 검증 솔루션

Python FastAPI Redis Azure(Databricks) Groq LPU
Problem (문제 상황)
  • 동시성 및 DB 병목: 다수의 사용자가 동일한 영상을 시청할 때 무거운 AI 파이프라인이 중복 요청되어 DB 병목 현상이 발생할 위험이 있었습니다.
  • 추론 지연: 기존 GPU 기반 추론 환경에서는 분석 완료까지 평균 22초 이상이 소요되어 실시간 개입에 치명적인 병목이 존재했습니다.
  • 환각 현상(Hallucination): 노이즈가 섞인 유튜브 자막을 그대로 주입할 경우 심각한 환각 현상과 불필요한 토큰 과금이 발생했습니다.
Solution (해결 과정)
  • Redis 분산 락 & Pub/Sub: 특정 video_id에 대한 분산 락(Lock)을 획득한 단 한 명의 워커만 파이프라인을 가동하고, 나머지는 결과를 구독(Pub/Sub)하며 대기하도록 설계했습니다.
  • 추론 엔진 교체: 토큰 생성 속도에 극단적으로 특화된 Groq LPU를 전격 도입하여 멀티 에이전트 환경의 I/O 대기 시간을 상쇄했습니다.
  • 데이터 정제 파이프라인: 메달리온 아키텍처(Bronze-Silver-Gold)를 구축하고, 정규표현식을 활용한 3중 필터링 및 스마트 청킹(Smart Chunking)을 적용하여 데이터를 고순도로 정제했습니다.
Result (핵심 성과)
  • 대규모 트래픽 발생 시에도 AI 연산 비용을 상수 O(1)로 고정하여 시스템 안정성을 확보했습니다.
  • 분석 소요 시간을 평균 22.4초에서 7.6초로 약 65% 단축하여 7초 내 실시간 차단 목표를 달성했습니다.
  • 시스템 전체 토큰 비용을 62% 이상 절감하였으며, 프로젝트 아키텍처 우수성을 인정받아 전체 1등을 달성했습니다.

2. RadiNow: 실시간 원전 데이터 감시 및 지능형 비상 대응 시스템

Python Flask Azure Functions Event Hubs PostgreSQL
Problem (문제 상황)
  • DB 커넥션 초과: 대시보드의 주기적인 상태 요청(Polling)으로 인해 다수 사용자 동시 접속 시 심각한 DB 병목 현상이 예상되었습니다.
  • 네트워크 낭비 및 지연: 비정기적인 정부 재난 문자를 잡기 위해 무의미한 Polling을 반복할 경우 리소스가 낭비되고 인지 지연(Latency)이 초래되었습니다.
  • 데이터 유실: 실시간 대량 수집되는 원전 센서 데이터를 직접 RDB에 적재할 경우 트래픽 스파이크 시 데이터 유실 위험이 존재했습니다.
Solution (해결 과정)
  • In-Memory 캐싱: APScheduler를 도입해 백그라운드 스레드가 일정 주기로 DB를 조회하고 그 결과를 메모리에 캐싱하도록 변경했습니다.
  • 단방향 스트리밍(SSE): 서버에서 신규 데이터 이벤트를 감지할 때만 클라이언트로 데이터를 푸시(Push)하는 SSE(Server-Sent Events) 아키텍처를 전면 도입했습니다.
  • 서버리스 & 버퍼링: Azure Event Hubs를 중간 버퍼로 도입하고, Azure Functions를 활용해 전처리와 ETA 계산 함수를 독립적으로 분리했습니다.
Result (핵심 성과)
  • 수백 건의 동시 Polling 요청에도 DB 접근 없이 O(1) 속도로 메모리에서 즉시 응답하도록 개선했습니다.
  • 무의미한 반복 요청 부하를 완전히 제거하고, 재난 문자가 화면에 즉시 렌더링되는 완벽한 실시간성을 확보했습니다.
  • 비상 상황 시 각 함수 단위로 독립적인 스케일 아웃(Scale-out)이 가능한 구조를 완성하여 프로젝트 평가 1등을 수상했습니다.

3. ZSE-SBIR (스케치 기반 이미지 검색 시스템)

Python PyTorch GCP (Deep Learning VM) ViT
Problem (문제 상황)
  • UI 프리징: 무거운 AI 모델 로딩 및 탐색 작업을 순차적으로 수행할 경우, 사용자 인터페이스(GUI)가 완전히 멈추는(Freezing) 병목 현상이 발생했습니다.
  • 검색 정확도 저하: 기존 모델이 Object-level에 치중되어 스케치와 사진 간의 크기 및 묘사 차이로 인해 정확도가 떨어지는 한계가 있었습니다.
Solution (해결 과정)
  • 병렬 처리: 관리 프로그램이 할당하는 작업을 n개의 쓰레드가 분담하도록 쓰레드 풀링(Thread Pooling) 병렬 연산 아키텍처를 도입했습니다.
  • GCP 대규모 재학습: 구글 클라우드 플랫폼의 Deep Learning VM을 활용하여, FS-COCO 데이터셋 10,000장(Train 9,000장, Valid 1,000장)으로 ViT 기반 모델 파인튜닝을 주도했습니다.
Result (핵심 성과)
  • 수백 장의 이미지 출력 연산 시 3초 미만이 소요되도록 하여 끊김 없는 원활한 UX를 보장했습니다.
  • 모델 재학습을 통해 스케치 기반 사진 검색의 거리 평균 정확도(유사도)를 0.89에서 0.93으로 대폭 개선했습니다.
